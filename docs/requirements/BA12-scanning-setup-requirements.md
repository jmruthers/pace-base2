## 2. Overview

BA12 delivers the event operator admin surface at `/scanning` for configuring scan points, downloading on-demand offline manifests, reviewing unresolved sync conflicts as a display-only consumer of BA14's conflict contract, and accessing scan-history records for the current event.

Operators can create scan points bound to a context (site, activity, transport, or meal) and direction, configure optional resource bindings, activate and deactivate points, launch the live scanning runtime for a chosen point, download JSON manifests per context type, view unresolved conflicts produced by BA14's sync processing, and review the full event scan history joined to participant and card identity.

This slice is the admin hub only. It does not own the scanning runtime (`/scanning/:scanPointId` — BA13), the sync and reconciliation logic (BA14), the tracking dashboard (BA16), or card lifecycle management (TEAM-owned). The mutation and read surfaces that depend on forward-spec tables are fully specified in this document and blocked from build execution until those tables and RLS policies land in dev-db.

## 3. What this slice delivers

### Purpose

Event staff need a single admin surface to set up and manage all scan points for an event before and during scanning operations — choosing what to scan (context type), which direction (in, out, or both), and which specific resource (activity session, transport leg) each point is watching. They also need to pull fresh offline manifests, review sync conflicts generated by BA14 without modifying them, and audit the full scan history.

### Surfaces

- **`/scanning`** — gated page presenting the scanning admin hub for the currently selected event. Contains four logical sections: scan-point management (DataTable with create/edit/deactivate/activate/launch actions), manifest downloads (one button per context type), conflict list (read-only DataTable of upload-conflict scan events), and scan history (read-only DataTable of all scan events for the event).

### Boundaries

- `/scanning/:scanPointId` is owned by BA13 (scanning runtime). BA12 navigates there; it does not render runtime content inline.
- Sync/reconciliation logic, offline queue state, upload retry rules, idempotency semantics, and conflict persistence rules belong to BA14. BA12 reads `base_scan_event` where `validation_result = 'upload_conflict'` as a display-only consumer of `BA14.conflict.contract`. BA12 does not define what constitutes a conflict, when conflicts appear, or how they are resolved.
- The tracking dashboard at `/scanning/tracking` belongs to BA16.
- Card issue, deactivation, and replacement workflows are TEAM-owned. BA12 reads `core_member_card.card_identifier` and `core_member_card.is_active` for manifest generation and history display only.
- Manifest generation is on-demand and client-side: no persisted manifest table exists in this slice.
- Device registry is excluded from MVP scope.

### Architectural posture

- All reads use **`useSecureSupabase()`** from `@solvera/pace-core/rbac`. No service-role client in route code.
- `base_scan_point` RLS uses the older `check_user_event_access` / `check_user_is_event_creator` pattern (confirmed against dev-db 2026-05-12), not `check_rbac_permission_with_context`. SELECT is permitted via `check_user_event_access`. INSERT and UPDATE are permitted via `check_user_is_event_creator`. DELETE is also permitted for event creators via `check_user_is_event_creator` (operators with creator rights can delete scan points — contrast with the specced "denied" posture; the live policy is more permissive). service_role bypasses all RLS. No RPC exists in dev-db for scan-point mutations.
- > **Route read access:** Enforced by the app authenticated shell / PaceAppLayout `routeAccessDenied` and [`base-route-registry.ts`](../../src/features/navigation/base-route-registry.ts). The page component must not wrap content in an outer `PagePermissionGuard operation="read"` unless this slice explicitly requires a **scoped read** override (`scope={{ organisationId, eventId, appId }}`).
- **`PagePermissionGuard`** gates **mutation affordances** (`create` / `update` / `delete`) per §10; route read is shell-owned only.
- Event context is resolved via **`useEvents()`** — use `selectedEvent.id` as `event_id` at all data boundaries.
- Manifest generation executes direct SELECT queries via the secure Supabase client — no RPC or Edge function.

### Page-level guards and evaluation ordering for `/scanning`

1. > **Route read access:** Enforced by the app authenticated shell / PaceAppLayout `routeAccessDenied` and [`base-route-registry.ts`](../../src/features/navigation/base-route-registry.ts). The page component must not wrap content in an outer `PagePermissionGuard operation="read"` unless this slice explicitly requires a **scoped read** override (`scope={{ organisationId, eventId, appId }}`).

2. **Route read access** is enforced by the authenticated shell / `PaceAppLayout` `routeAccessDenied` and [`base-route-registry.ts`](../../src/features/navigation/base-route-registry.ts). If access is denied, **`AccessDenied`** is rendered immediately. No-event messaging does not replace or precede denial.
3. If the guard is loading and no custom `loadingFallback` prop is supplied, **`PagePermissionGuard`** renders `null` — neither children nor the denial state is shown. A null Supabase client (transient auth initialisation) renders a centred **`LoadingSpinner`** in the main content region.
4. If the guard permits and **no event is selected** (`selectedEvent` is null or `selectedEvent.id` is falsy), the page shows a blocking **`Card`** instructing the user to select an event in the shell header. Data fetches do not run. The four section DataTables are not rendered.
5. If the guard permits and **an event is selected**, all four sections load for that `event_id`.

## 4. Functional specification

Prefix legend: **`SC`** /scanning page-level, **`SP`** scan-point management, **`MF`** manifest, **`CF`** conflict review, **`HI`** scan history.

### `/scanning` — page entry

1. **SC-PE-01 —** Navigating to `/scanning` renders inside the BA00 authenticated shell with no mandatory query parameters.
2. **SC-PE-02 —** With an event selected, the page loads all `base_scan_point` rows for `selectedEvent.id` and `organisation_id`, all conflict-flagged `base_scan_event` rows, and the full scan-event history for the event — all in parallel.
3. **SC-PE-03 —** Page chrome: `h1` "Scanning Setup" and a subtitle naming the current event and describing the surface ("Configure scan points and review scanning activity for this event.").
4. **SC-TR-01 —** A **"View Tracking Dashboard"** `Button variant="outline"` is placed in the heading block, right-aligned alongside the `h1` and subtitle. Pressing it navigates to `/scanning/tracking` (BA16). The button is only rendered when the authenticated user has `read:page.scanning`. It is rendered regardless of whether an event is selected — the tracking dashboard handles its own event-selection guard.

### `/scanning` — loading states

4. **SC-LS-01 —** While any of the three data queries is unresolved and the guard permits with an event selected, the corresponding `DataTable` receives `isLoading` true (table shows built-in loading row with `LoadingSpinner` per pace-core `DataTable` behaviour). Each section loads independently.
5. **SC-LS-02 —** A null Supabase client (transient auth initialisation) renders a centred `LoadingSpinner` in the main content region before any section renders.

### `/scanning` — empty states

6. **SC-ES-01 —** Event selected, zero scan points: the scan-point `DataTable` shows its empty state with title "No scan points configured" and description "No scan points have been configured for this event." A **"Create scan point"** `Button variant="default"` is rendered below the empty state copy (only when user has `create:page.scanning`).
7. **SC-ES-02 —** Event selected, no upload conflicts: the conflict `DataTable` shows its empty state with description "No unresolved sync conflicts." No CTA is rendered.
8. **SC-ES-03 —** Event selected, no scan events: the history `DataTable` shows its empty state with description "No scan events recorded yet." No CTA is rendered.
9. **SC-ES-04 —** No event selected: the page shows a blocking `Card` below the heading block with `CardTitle` "No event selected" and `CardDescription` "Select an event from the header to configure scanning." Data fetches do not run; all four sections are absent.

### `/scanning` — error states

10. **SC-ER-01 —** Scan-point list fetch failure: `Alert variant="destructive"` with the normalised error message from `normalizeSupabaseError` and a `Button variant='outline' size='small'` labelled "Retry" (placed below the `AlertDescription` within the same `Alert`) that refetches the scan-point query.
11. **SC-ER-02 —** Conflict list fetch failure: `Alert variant="destructive"` with normalised error and a `Button variant='outline' size='small'` labelled "Retry" (placed below the `AlertDescription` within the same `Alert`) for the conflict query.
12. **SC-ER-03 —** History list fetch failure: `Alert variant="destructive"` with normalised error and a `Button variant='outline' size='small'` labelled "Retry" (placed below the `AlertDescription` within the same `Alert`) for the history query.

### `/scanning` — scan-point list content

13. **SP-PC-01 —** The scan-point list is a **`DataTable`** with title "Scan Points".
14. **SP-PC-02 —** Columns (in order): **Name** (`scan_point.name` read directly from the DB, sortable), **Context** (`Badge`, sortable), **Direction** (`Badge`, sortable), **Resource** (text or "—", sortable), **Status** (paired badges: active/inactive status + offline-capable indicator, sortable by status), **Actions** (Launch — icon-only row control with `aria-label="Launch scan point"`, Edit, Deactivate/Activate).
15. **SP-PC-03 —** Rows are ordered by Name ascending on initial load.
16. **SP-PC-04 —** The `DataTable` enables search (across Name and resource display text), sorting, and pagination. Filtering, export, import, bulk editing, creation via DataTable toolbar, and deletion via DataTable toolbar are disabled — mutations use custom row actions and a dedicated "Create scan point" button above the table.

### `/scanning` — create scan point action

17. **SP-PA-01 —** A **"Create scan point"** `Button variant="default"` is placed immediately above the scan-point `DataTable` (outside the DataTable toolbar). The button is only rendered when the authenticated user has `create:page.scanning`. Pressing it opens the create dialog (see SP-PA-02 to SP-PA-05).
18. **SP-PA-02 —** The create dialog is a controlled `Dialog` containing a `Form`. `DialogTitle` "Create scan point". Fields in `DialogBody` (in order):
    - **Name** — required text input, `FormField` `name="name"` `label="Name"` `required`, max 100 characters, placeholder "e.g. Main gate, Activity hub entrance", helper text "A short label to identify this scan point." Error copy: "Name is required." when empty; "Name must be 100 characters or fewer." when over max length.
    - **Context type** — required `Select` with options: "Site" (`site`), "Activity" (`activity`), "Transport" (`transport`), "Meal" (`meal`). No other values accepted.
    - **Direction** — required `Select` with options: "In" (`in`), "Out" (`out`), "Both" (`both`), "Neutral" (`neutral`).
    - **Resource** — optional text display control when context type is `site` or `meal` (no resource selection shown). Required `Select` when context type is `activity` (select from `base_activity_session` rows for the current event). Required `Select` when context type is `transport` (select from `trac_activity` rows for the current event — `trac_itinerary_assignment` is live so transport context is fully enabled). See BR-SP-01.
19. **SP-PA-03 —** `DialogFooter` contains `Button variant="outline"` labelled "Cancel" (closes the dialog, no data submitted) and `Button variant="default"` `type="submit"` labelled "Create scan point".
20. **SP-PA-04 —** On submit: validate form per BR-V-01. Success: insert into `base_scan_point`, close dialog, toast `variant='success'` "Scan point created", scan-point list refreshes. Failure (validation): inline field error per BR-V-01. Failure (save): dialog stays open, toast `variant='destructive'` with normalised error.
21. **SP-PA-05 —** New scan points are created with `is_active = true`.

### `/scanning` — edit scan point action

22. **SP-PA-06 —** An **Edit** row action appears on active scan-point rows. Edit is not rendered on inactive rows (see Q-D2 resolution). Pressing Edit opens the edit dialog pre-populated with the scan point's current `name`, `context_type`, `direction`, and `resource_id`.
23. **SP-PA-07 —** The edit dialog is a controlled `Dialog` with `DialogTitle` "Edit scan point". Fields in `DialogBody` are the same set as the create dialog (name, context type, direction, resource), pre-populated with the existing values. Resource binding clears when context type is changed.
24. **SP-PA-08 —** `DialogFooter` contains `Button variant="outline"` "Cancel" (closes dialog) and `Button variant="default"` `type="submit"` "Save changes".
25. **SP-PA-09 —** On submit: validate form per BR-SP-02 and BR-V-01. Success: update `base_scan_point` row (context type, direction, resource type, resource id only), close dialog, toast `variant='success'` "Scan point updated", scan-point list refreshes. Failure: dialog stays open, toast `variant='destructive'` with normalised error.
26. **SP-PA-10 —** Editing a scan point does not affect any `base_scan_event` rows referencing that scan point. Scan history is immutable and unaffected by edits.

### `/scanning` — deactivate scan point action

27. **SP-PA-11 —** A **Deactivate** row action appears on active scan-point rows. Pressing Deactivate opens a confirmation `Dialog` with:
    - `DialogTitle` "Deactivate scan point"
    - `DialogBody`: `scan_point.name` of the scan point rendered as `<p className='font-semibold text-sm text-foreground'>`, and copy: "Deactivating this scan point will remove it from the live scanning list. Existing scan history will not be affected."
    - `DialogFooter`: `Button variant="destructive"` "Deactivate" + `Button variant="outline"` "Cancel".
28. **SP-PA-12 —** On confirm: set `is_active = false` on the `base_scan_point` row. Success: dialog closes, toast `variant='success'` "Scan point deactivated", list refreshes. Failure: dialog stays open, toast `variant='destructive'` with normalised error.
29. **SP-PA-13 —** The row is not deleted. Scan history referencing the point remains visible in the history section.

### `/scanning` — activate scan point action

30. **SP-PA-14 —** An **Activate** row action appears on inactive scan-point rows (the Deactivate action is not rendered on inactive rows). Pressing Activate does not open a dialog — it immediately sets `is_active = true`. Success: toast `variant='success'` "Scan point activated", list refreshes. Failure: toast `variant='destructive'` with normalised error.
31. **SP-PA-15 —** After activation, the Edit row action becomes available again on that row.

### `/scanning` — launch action

32. **SP-PA-16 —** A **Launch** row action (icon-only control, `aria-label="Launch scan point"`) appears on all scan-point rows regardless of active/inactive status. Pressing it navigates to `/scanning/:scanPointId` where `:scanPointId` is the `base_scan_point.id`. Navigation is unconditional — BA13 handles all runtime validation including inactive point warnings.

### `/scanning` — manifest downloads

33. **MF-PA-01 —** A "Manifests" section is rendered below the scan-point `DataTable`. It is not a DataTable; it is a `Card` containing a download button for each supported context type.
34. **MF-PA-02 —** Four buttons appear (one per context type): "Download Site Manifest", "Download Activity Manifest", "Download Transport Manifest", "Download Meal Manifest". All four buttons are rendered regardless of whether scan points of those types exist. Buttons are only rendered when the user has `read:page.scanning`.
35. **MF-PA-03 —** Pressing a manifest download button executes the appropriate manifest query (see §6 BR-M-01 to BR-M-04), serialises the result as a JSON blob, and triggers a browser file download with a filename of the form `{context-type}-manifest-{event-id}-{YYYY-MM-DD}.json`. No server round-trip beyond the Supabase SELECT query; no intermediate storage.
36. **MF-PA-04 —** Manifest payload shape: `[{ card_identifier: string, person_id: uuid, name: string }]`. One file per button press.
37. **MF-PA-05 —** While a manifest query is executing, the corresponding button shows a `LoadingSpinner` inline and is `disabled`. Other buttons remain interactive.
38. **MF-PA-06 —** If a manifest query returns zero rows, the download proceeds with an empty array (`[]`). No error toast is shown; the empty file is a valid result.
39. **MF-PA-07 —** If a manifest query fails, a toast `variant='destructive'` is shown with the normalised error message. No file download occurs.
40. **MF-PA-08 —** The "Download Transport Manifest" button is enabled. `trac_itinerary_assignment` is confirmed live in dev-db (`rkytnffgmwnnmewevqgp`) and `core_member_card` is live; the transport manifest query (BR-M-03) is buildable. No disabled state applies.

### `/scanning` — conflict review

41. **CF-PA-01 —** A "Sync Conflicts" section is rendered below the manifests `Card`. It is a `DataTable` containing `base_scan_event` rows where `validation_result = 'upload_conflict'`, scoped to `selectedEvent.id`. BA12 displays these rows as produced by BA14's sync processing — BA12 does not define what triggers them or how they are resolved.
42. **CF-PA-02 —** Pressing **View detail** on a conflict row opens a read-only `Dialog` with `DialogTitle` "Conflict detail". See SP-CF-Detail in §5 for the field layout.
43. **CF-PA-03 —** The conflict detail dialog has no edit or resolve affordance. All fields are display-only.
44. **CF-PA-04 —** Column set and sort for the conflict list: see CF-PC-01 below and §6 BR-CR-01.

45. **CF-PC-01 —** Conflict list columns (in order): **Scan point** (`scan_point.name` from the joined `base_scan_point` row, sortable), **Scanned at** (formatted `scanned_at` timestamptz, sortable), **Card identifier** (`core_member_card.card_identifier` via `scan_card_id` join, sortable), **Original reason** (`validation_reason` text, sortable), **Synced at** (formatted `synced_at` timestamptz, sortable), **Actions** (View detail). Rows ordered by `scanned_at` descending on initial load.

### `/scanning` — scan history

46. **HI-PA-01 —** A "Scan History" section is rendered below the conflict list. It is a `DataTable` showing all `base_scan_event` rows for `selectedEvent.id`, joined to scan point, card, and participant identity.
47. **HI-PC-01 —** History columns (in order): **Scan point** (`scan_point.name` from the joined `base_scan_point` row, sortable), **Participant** (name from the `base_application` join — preferred name or first/last name, sortable), **Card identifier** (`core_member_card.card_identifier`, sortable), **Result** (`Badge`, sortable), **Reason** (`validation_reason` text or "—", sortable), **Scanned at** (formatted `scanned_at`, sortable), **Synced at** (formatted `synced_at` or "—", sortable). Rows ordered by `scanned_at` descending.
48. **HI-PA-02 —** No actions are available on history rows. History is read-only — no delete, edit, or override affordance.

### Both sections — permission-conditional rendering

49. **SC-PERM-01 —** When the authenticated user has `read:page.scanning` but lacks `create:page.scanning`, the "Create scan point" button above the DataTable and within empty state is not rendered.
50. **SC-PERM-02 —** When the user lacks `update:page.scanning`, the Edit, Deactivate, and Activate row actions are not rendered.
51. **SC-PERM-03 —** When the user lacks `read:page.scanning` entirely, the route renders `AccessDenied`.

### Non-empty verification

52. **SC-VER-01 —** Non-empty state verification uses BA18 seed data, not route-local fixtures.

## 6. Business rules

### BR-SP-Label — Scan-point display label

**Trigger:** any surface that must display a human-readable identifier for a `base_scan_point` row.

**Rule:** The scan point display label is `scan_point.name` as stored in the database. No client-side derivation is required.

Context type, direction, and resource continue to appear as separate Badge columns or fields in their own right — they are not concatenated into a display label.

### BR-SP-01 — Create scan point

**Input fields:** `name` (text, required, max 100 characters), `event_id` (`selectedEvent.id`), `organisation_id` (from event context), `context_type` ∈ {`site`, `activity`, `transport`, `meal`}, `direction` ∈ {`in`, `out`, `both`, `neutral`}, optional `resource_type`, optional `resource_id`.

**Resource binding rules:**
- `site` and `meal` contexts: `resource_type` and `resource_id` are null. Resource selector not shown.
- `activity` context: `resource_type = 'activity_session'`, `resource_id` = `base_activity_session.id` — required.
- `transport` context: `resource_type = 'trac_activity'`, `resource_id` = `trac_activity.id` — required.

**Insert behaviour:** row is created with `is_active = true`. `created_by = auth.uid()`, `created_at = now()`.

**Write contract:** direct client-side INSERT into `base_scan_point` via the RBAC-checked RLS policy for `insert:page.scanning`. No RPC in dev-db.

**Failure class:** `validation_error.scan_point_resource_binding_invalid` when resource-bound context is submitted without a valid `resource_id`, or when `resource_id` does not resolve to a live record.

### BR-SP-02 — Edit scan point

**Editable fields:** `name`, `context_type`, `direction`, `resource_type`, `resource_id` only.

**Edit is not available on inactive scan points.** The Edit row action is absent on rows where `is_active = false`.

**Effect on history:** updating a scan point row does not mutate any `base_scan_event` rows referencing that scan point. Scan history displays `scan_point.name` at read time from the current (updated) scan-point row — there is no snapshotted name on `base_scan_event`.

**Write contract:** direct client-side UPDATE on `base_scan_point` via the RBAC-checked RLS policy for `update:page.scanning`.

### BR-SP-03 — Deactivate scan point

**Mutation:** sets `is_active = false` on the `base_scan_point` row. The row is not deleted.

**Historical scan events referencing the deactivated point remain intact** and remain visible in the history section.

**State after deactivation:** the row moves to inactive status in the scan-point list. The Activate row action replaces the Deactivate action. The Edit row action is removed.

**Write contract:** direct client-side UPDATE on `base_scan_point.is_active` column via the RBAC-checked RLS policy for `update:page.scanning`.

### BR-SP-04 — Activate scan point

**Mutation:** sets `is_active = true` on the `base_scan_point` row.

**No confirmation dialog** — activation is not a destructive action.

**State after activation:** the row returns to active status. The Edit and Deactivate row actions become available again.

**Write contract:** same RLS update policy as BR-SP-03.

### BR-M-01 — Site manifest query

**Trigger:** operator presses "Download Site Manifest".

**Query:** `base_application` WHERE `status = 'approved'` AND `event_id = selectedEvent.id` AND `organisation_id = selectedOrganisation.id`, JOINed to `core_member` WHERE `core_member.person_id = base_application.person_id` AND `core_member.organisation_id = selectedOrganisation.id`, then JOINed to `core_member_card` WHERE `core_member_card.member_id = core_member.id` AND `core_member_card.is_active = true`.

**Output payload per row:** `{ card_identifier: core_member_card.card_identifier, person_id: base_application.person_id, name: <name> }` where `name` = `preferred_name` if `preferred_name` is non-null and non-empty string; otherwise `first_name || ' ' || last_name`. Source columns on `core_person`: `preferred_name`, `first_name`, `last_name`.

**Exclusions:** coordinators or staff who do not have an approved `base_application` row are excluded. No application = not in manifest.

**Source table status:** `base_application` is in dev-db. `core_member_card` is forward spec (TEAM-owned). This manifest type is partially gated — the `base_application` query can be verified; the card join requires `core_member_card` to land.

### BR-M-02 — Activity manifest query

**Trigger:** operator presses "Download Activity Manifest".

**Query:** `base_activity_booking` WHERE `status = 'confirmed'` AND `event_id = selectedEvent.id`, JOINed to `base_activity_session` (via `session_id`). The join path from `base_activity_booking` to `core_person` is: `base_activity_booking.application_id → base_application.id → base_application.person_id`. The FK column on `base_activity_booking` is `application_id uuid`. Conceptual PostgREST join chain: `base_activity_booking!inner(application_id, base_application!inner(person_id, core_person!inner(preferred_name, first_name, last_name)))`. JOINed to `core_member` WHERE `core_member.person_id = base_application.person_id` AND `core_member.organisation_id = selectedOrganisation.id`, then JOINed to `core_member_card` WHERE `core_member_card.member_id = core_member.id` AND `core_member_card.is_active = true`.

**Output payload per row:** `{ card_identifier: core_member_card.card_identifier, person_id: base_application.person_id, name: <name> }` where `name` = `preferred_name` if `preferred_name` is non-null and non-empty string; otherwise `first_name || ' ' || last_name`. Source columns on `core_person`: `preferred_name`, `first_name`, `last_name`.

**Build prerequisite:** `base_activity_booking`, `base_activity_session` (forward spec per BA09/BA11) and `core_member_card` (forward spec, TEAM-owned) must land before this manifest type is buildable.

### BR-M-03 — Transport manifest query

**Trigger:** operator presses "Download Transport Manifest".

**Query:** `trac_itinerary_assignment` WHERE `event_id = selectedEvent.id` AND `organisation_id = selectedOrganisation.id` AND `resource_type = 'transport'`, JOINed via `application_id → base_application.id → base_application.person_id → core_person` (for name), then JOINed to `core_member` WHERE `core_member.person_id = base_application.person_id` AND `core_member.organisation_id = selectedOrganisation.id`, then JOINed to `core_member_card` WHERE `core_member_card.member_id = core_member.id` AND `core_member_card.is_active = true`.

**Critical join note:** `trac_itinerary_assignment.application_id` is the participant join column (FK → `base_application.id`). There is no `person_id` column on `trac_itinerary_assignment`. The join chain is: `trac_itinerary_assignment.application_id → base_application.id → base_application.person_id → core_person.id` and separately `→ core_member.person_id → core_member.id → core_member_card.member_id`.

**Output payload per row:** `{ card_identifier: core_member_card.card_identifier, person_id: base_application.person_id, name: <name> }` where `name` = `preferred_name` if `preferred_name` is non-null and non-empty string; otherwise `first_name || ' ' || last_name`. Source columns on `core_person`: `preferred_name`, `first_name`, `last_name`.

**Build status:** `trac_itinerary_assignment` and `core_member_card` are both confirmed live in dev-db (`rkytnffgmwnnmewevqgp`). This manifest type is fully buildable.

### BR-M-04 — Meal manifest query

**Trigger:** operator presses "Download Meal Manifest".

**Query:** identical to BR-M-01 — approved `base_application` rows joined to active `core_member_card` for the event. Meal eligibility in MVP is event participation (approved application), not a dedicated meal booking.

**Build prerequisite:** same gating as BR-M-01 — `core_member_card` must land before the card join is buildable.

### BR-CR-01 — Conflict list read

**Query:** `base_scan_event` WHERE `validation_result = 'upload_conflict'` AND `scan_point_id IN (<event scan point IDs>)` (two-step pattern per §7.1), JOINed to `base_scan_point` (for `name`), JOINed to `core_member_card` (via `scan_card_id`, for `card_identifier`).

**Sorted:** `scanned_at` descending.

### BR-H-01 — Scan history read

**Query:** `base_scan_event` scoped to scan points for `selectedEvent.id` (two-step pattern per §7.1), JOINed to `base_scan_point` (for `name`), JOINed to `core_member_card` (via `scan_card_id`, for `card_identifier`), JOINed to `base_application` and `core_person` (for participant name — optional join, may return null).

**Participant name derivation:** `name` = `preferred_name` if `preferred_name` is non-null and non-empty string; otherwise `first_name || ' ' || last_name`. Source columns on `core_person`: `preferred_name`, `first_name`, `last_name`.

**Sorted:** `scanned_at` descending.

**All rows are read-only.** No edit, override, or delete affordance on history rows.

### BR-C-01 — Card lifecycle boundary

BA12 reads `core_member_card.card_identifier` and `core_member_card.is_active` for manifest generation and history display only. BA12 must not surface or navigate to any card issue, deactivation, or replacement workflow. Card management is TEAM-owned.

### 7.1 Read contracts — RBAC-checked RLS

**`base_scan_point` RLS (confirmed live in dev-db 2026-05-12 — uses older pattern):**

```sql
-- SELECT (authenticated): permitted via check_user_event_access

-- UPDATE (authenticated): permitted via check_user_is_event_creator
```

```sql
-- INSERT (authenticated): currently permitted via check_user_event_access
--   *** MUST BE CHANGED TO false BEFORE BA14 FLUSH PATH IS BUILT ***
--   (BA14 requires service_role-only INSERT; authenticated INSERT must be denied)
-- UPDATE (authenticated): DENIED
-- DELETE (authenticated): DENIED
-- service_role: ALL
```

```ts
from('base_scan_event')
```

### §9.2 Slice-specific caveats

**`DialogPortal` required for all dialogs**

Dialogs in BA12 must be mounted via `DialogPortal` outside the component tree, matching the pattern used in the pace-core2 DataTable source. Failing to use `DialogPortal` causes stacking context issues when dialogs are triggered from within DataTable rows.

**`useSecureSupabase()` returning null**

If `useSecureSupabase()` returns null (transient auth client initialisation), render a centred `LoadingSpinner` instead of issuing queries. Do not show an error state; the client resolves once auth state settles.

**`selectedEvent.id` — canonical field name**

`selectedEvent.id` is the canonical event identifier from `useEvents()`. Do not use `selectedEvent.event_id` — that field does not exist on `EventStub`.

**`Select` inside Dialog — no extra `positionMode` config needed**

`Select` `positionMode` auto-detects the Dialog context and uses `fixed` positioning. No manual override is needed for selectors inside the create and edit dialogs.

**Conflict list read — never mutate**

The conflict list query touches `base_scan_event`. BA12 must never issue an UPDATE, INSERT, or DELETE against `base_scan_event`. The table is write-only from BA13 (via BA14 sync). BA12 is a read-only consumer. Do not add any mutation path — even "mark resolved" — without an explicit BA14 contract change.

**Manifest download — browser blob URL pattern**

Use `URL.createObjectURL(new Blob([JSON.stringify(rows)], { type: 'application/json' }))` with a programmatic anchor element `click()` for the download trigger. Call `URL.revokeObjectURL(url)` after the click to avoid memory leaks. Do not route manifest data through a server endpoint or Edge function.

## 11. Acceptance criteria

- Given a user with `read:page.scanning` and an event selected, when `/scanning` loads, then the page renders with the "Scanning Setup" heading, the selected event name in the subtitle, and all four sections (scan points, manifests, conflicts, history).
- Given a user with `read:page.scanning` and no event selected, when `/scanning` loads, then a blocking Card "No event selected" is shown and no data sections render.
- Given a user without `read:page.scanning`, when navigating to `/scanning`, then `AccessDenied` is rendered and no scanning content is shown.
- Given a Supabase client null state, when `/scanning` loads, then a centred `LoadingSpinner` is shown in the main content region with no error state.
- Given an event with no scan points, when the scan-point DataTable loads, then the empty state "No scan points have been configured for this event." is shown with a "Create scan point" CTA (when user has `create:page.scanning`).
- Given a user without `create:page.scanning`, when the scan-point DataTable loads in the empty state, then the "Create scan point" CTA is absent.
- Given a user with `create:page.scanning` presses "Create scan point", when the dialog opens, then name, context type, direction, and conditionally resource fields are shown with no pre-populated values.
- Given the create form is submitted with `context_type = 'activity'` and no resource selected, when the form is validated, then an inline error "A resource is required for this context type." appears on the Resource field and no insert is attempted.
- Given a valid create form is submitted, when the insert succeeds, then the dialog closes, a success toast "Scan point created" appears, and the scan-point list refreshes with the new row.
- Given an active scan-point row, when the user with `update:page.scanning` presses Edit, then the edit dialog opens pre-populated with the row's name, context type, direction, and resource binding.
- Given an inactive scan-point row, then the Edit row action is absent regardless of user permissions.
- Given an active scan-point row, when the user presses Deactivate and confirms in the dialog, then the row's `is_active` becomes false, the list refreshes, and the Activate row action appears in its place.
- Given an inactive scan-point row, when the user with `update:page.scanning` presses Activate, then no dialog is shown, the row's `is_active` becomes true, and the Edit and Deactivate actions become available on that row.
- Given any scan-point row, when the user activates the **Launch** control (`aria-label="Launch scan point"`), then the browser navigates to `/scanning/:scanPointId`.
- Given a user with `read:page.scanning` presses any manifest download button (site, activity, transport, or meal), when the query runs, then the button shows a spinner and is disabled during execution, and a JSON file is downloaded after success.
- Given the manifest query returns zero rows, when the download completes, then a JSON file containing an empty array `[]` is downloaded with no error toast.
- Given the manifest query fails, when the failure is caught, then a `toast variant='destructive'` is shown with the normalised error and no file is downloaded.
- Given the scan-point list has rows with `context_type = 'site'` and `direction = 'both'`, then the Name cell displays `scan_point.name`, the Context cell shows the "Site" badge, the Direction cell shows the "Both" badge, and both the active/inactive badge and the `Badge variant="solid-acc-normal"` "Offline" badge appear in the Status cell.
- Given the conflict list has no rows with `validation_result = 'upload_conflict'`, then the conflict DataTable shows "No unresolved sync conflicts." with no CTA.
- Given the conflict list has upload-conflict rows, when the user presses "View detail" on a row, then the conflict detail dialog opens showing scan point label, card identifier, result, original reason, scanned at, synced at, notes, and override by — all read-only with no edit affordance.
- Given the scan history DataTable, when a row has `validation_result = 'accepted'`, then the Result cell shows `Badge variant="solid-main-normal"` "Accepted".
- Given a scan-point list fetch failure, then an `Alert variant="destructive"` with a Retry control appears in place of the scan-point DataTable.
- Given a user without `update:page.scanning`, then the Edit, Deactivate, and Activate row actions are absent from all scan-point rows.
- Given a scan point with `context_type = 'activity'`, a resolved resource name 'Rock Climbing', and `direction = 'in'`, when the scan-point list loads, then the Name cell displays `scan_point.name`, the Context badge shows "Activity", the Direction badge shows "In", and the Resource cell shows "Rock Climbing".

## 14. Build execution rules

**All four section DataTables may be built independently of each other.** The scan-point DataTable's read surface builds as soon as `base_scan_point` RLS is deployed. Conflict and history sections build as soon as `base_scan_event` and `core_member_card` land. Manifest buttons for site and meal build as soon as `core_member_card` lands. Activity manifest and transport manifest remain blocked on their respective forward-spec tables.

Specific blocking conditions:

- **Stop** on conflict list and scan history build until the `base_scan_event` authenticated INSERT RLS is changed to `false`. (The SELECT surface is already readable under the current RLS; only the INSERT change is blocking BA14's flush path.)
- **Do not** issue any UPDATE, INSERT, or DELETE against `base_scan_event` from BA12 code. All scan-event writes are BA13/BA14 concerns.
- **Do not** issue a DELETE against `base_scan_point`. Deactivation is `is_active = false` only.
- **Do not** use `window.confirm()` for any confirmation flow.
- **Do not** ship hardcoded fixture scan-point or scan-event rows in route code. Loading, empty, and error states handle no-data conditions.
- **Do not** route manifest data through an Edge function or server-side endpoint. Manifests are client-side SELECT queries with browser file download.
- **Do not** implement device registry, offline queue state tracking, upload retry logic, or conflict resolution in this slice.
